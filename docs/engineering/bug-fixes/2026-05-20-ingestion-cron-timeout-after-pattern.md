# Ingestion cron timeout — decouple cron-job.org wall via Next.js `after()` — Bug-Fix Record

## Summary
- Problem addressed: `/api/cron/fetch-editorial-inputs` failed `(timeout)` at cron-job.org's 30-second external HTTP wall on 2026-05-19 (both 10:15 UTC and 11:45 UTC scheduled runs). The 12:15 UTC health check correctly returned HTTP 500 because the Editorial Queue had fewer than the 7-row threshold. Sentry surfaced the same condition as `Cron failure: rss-feed-refresh` (the RSS check-in stopped completing) and as a related noise stream `RssError: Feed request retry exhausted for Reuters...`. User trials were imminent; ingestion had been broken for ≥24 hours.
- Root cause: The cron handler had grown into a three-stage sequential pipeline — `runNewsletterIngestion` → `runDailyNewsCron` → `runEditorialStaging` — whose p50 latency sat right against the 30-second cron-job.org external HTTP timeout. The largest recent additions were the Phase 4.5 source-circuit-breaker (PR #251), which adds three Notion API calls per RSS source inside the fetch path, and the editorial-staging Notion writer (PR #237), which writes seven candidates sequentially with two Notion calls each. On any day with normal Notion latency variance, the handler exceeded 30 s; cron-job.org disconnected; the rest of the pipeline never completed; the in-flight Sentry check-in was never closed. Reuters retry-exhaustion (`feeds.reuters.com` has been DNS-dead since ~2021) was a contributing noise source but **not** the timeout driver — DNS for that host fails in roughly 8 ms locally, consuming sub-second wall time even with retries.
- Affected object level: Card (Editorial Queue rows in Notion), with upstream impact on Signal generation (no `signal_posts` written) and Surface Placement (no rows available for editorial review).

## Fix
- **Exact change:** Convert `GET /api/cron/fetch-editorial-inputs` to fire-and-forget using Next.js's stable `import { after } from "next/server"` API. After auth gating, the handler returns HTTP 202 in <1 s and schedules the three-stage pipeline to run via `after()`. Vercel keeps the function alive up to the `maxDuration: 60` budget declared in `vercel.json` to complete the `after()` work, so the pipeline still gets its full execution window without making cron-job.org wait. The 30-second external HTTP wall no longer applies. Health-check observability at 12:15 UTC remains the authoritative ingestion-success signal.
  - Inside the `after()` callback, the pipeline is wrapped in `runWithInternalTimeout(stageRef, runIngestionPipeline(stageRef), 55_000)`. A `stageRef` cursor is advanced as the pipeline crosses stages (`newsletter` → `rss` → `editorial_staging`). On internal-timeout, an `IngestionTimeoutInternalError` is thrown and captured to Sentry with tags `{ route, stage, failure_type: "ingestion_timeout_internal" }` and extra `{ internalTimeoutMs }`, then `Sentry.flush(2_000)` is awaited so the event ships before Vercel freezes the function. Without this guard we would trade visible 30-second cron-job.org failures for silent 60-second function-level kills — the guard makes timeouts loud again, and attributes the wall to a named stage. The 55-second internal wall sits below the 60-second Vercel function-level kill so this guard always fires first.

## Remaining Risks / Follow-up
- **Reuters dead-feed cleanup** is confirmed but intentionally scoped out of this PR. Follow-on PR title: `chore(rss): remove dead Reuters feeds from active source manifest`. Scope: remove `horizon-reuters-world` and `horizon-reuters-business` from `DEFAULT_DONOR_FEED_IDS` in `src/adapters/donors/registry.ts`; remove the corresponding feed entries from `src/adapters/donors/horizon/index.ts`; review and update any reference in `docs/audits/` or the public source manifest doc that still claims Reuters is active. Cross-link this bug-fix record from that PR's description so the operational chain is recoverable.
- **Two cron firings overlap if the first runs >90 minutes.** With a 60-second `maxDuration` wall this is structurally impossible today, but if the function ever grows past 90 minutes of work (e.g., a future LLM-heavy enrichment stage), we should add a `briefingDate`-keyed lock before the pipeline body. Not building it now — it would be speculative.
- **Failure observability is asymmetric.** With the `after()` pattern, the cron-job.org dashboard will show every run as 202 Success regardless of actual ingestion outcome. Authoritative success signals are now (a) Sentry (timeout / unexpected error capture), (b) Notion Pipeline Log row, (c) the 12:15 UTC `/api/cron/health` check. This is explicitly accepted; the health check already exists and was built for exactly this purpose. If cron-job.org's email notification on non-2xx was previously the primary alarm, it has been silently downgraded — the 12:15 UTC health check now carries that alarm contract.
- **Internal-timeout granularity is at the stage boundary, not within a stage.** If the newsletter stage hangs forever, the timeout still fires at 55 seconds with `stage="newsletter"`. If a single Notion call inside editorial-staging hangs, the timeout fires with `stage="editorial_staging"` but doesn't tell us *which* candidate was being written. Acceptable for v1; revisit if Sentry shows clustering on a single stage without finer detail being useful.
- **Stage-attribution Sentry coverage is single-test.** The unit test `captures IngestionTimeoutInternal to Sentry when the pipeline exceeds the internal budget` only asserts the path where `stageRef.current === "newsletter"` (the first stage). The "rss" and "editorial_staging" paths are exercised by the same `stageRef` mutation logic and the same `Promise.race` wiring, so the wiring itself is verified; what is *not* verified by a test is that the Sentry tag carries the right value when the timer fires mid-`rss` or mid-`editorial_staging`. The mutation sites are dead-simple assignments at `route.ts:111, 118, 121`, so the gap is low-risk — but if a future change moves the mutation off the function-local cursor (e.g., refactors to a closure pattern) the regression would not be caught. Add two parametrised tests if Sentry shows stage-attribution drift on a real production timeout.
- **The cron reschedule to 1×/day at 20:00 Taipei** (the prompt's Step 8) is held until this fix is observed working on at least one scheduled run. That sequencing is intact.
