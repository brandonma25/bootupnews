# ADR 001: Public Source Manifest

The decision is to adopt a public surface source manifest as the enforced source of truth for public surface source selection. The first manifest surface is `public.home`, and it resolves ordered demo-source IDs into the `Source[]` list used by the unauthenticated public dashboard path. This is intentionally narrower than a full source-system redesign: it gives the public product one explicit activation layer without changing the donor registry, the source catalog, the ingestion resolver, ranking, clustering, or the legacy MVP default helper.

The context comes from the runtime source activation audit (`../audits/source-activation-audit.md`) and the source architecture audit (`../audits/source-architecture-audit.md`). The activation audit showed that runtime source truth is split across MVP public defaults, donor defaults, and probationary runtime feeds. It also showed that Reuters World is already donor-live as `horizon-reuters-world`, but not present in the public MVP source set that reaches the unauthenticated Home path. The architecture audit traced the public Home route through `getDashboardPageState("/")`, `getDashboardData`, `getPipelineBackedDashboardData`, and `generateDailyBriefing`, and found that the public supplied-source path bypasses the no-argument donor defaults where Reuters World is otherwise available.

The architecture audit considered three paths. Option 1 was a runtime source plan resolver that would unify public, category, donor, probationary, and personalized fallback plans behind one resolver. Its advantage is conceptual cleanliness and a better long-term activation model, but it has a wider change surface and higher regression risk because it touches multiple runtime boundaries. Option 2 was to make the donor registry the executable runtime truth and treat the catalog as a metadata overlay. Its advantage is alignment with existing donor ingestion, but it requires careful source identity normalization between demo IDs and donor IDs. Option 3 was to add category or surface manifests on top of the existing lists. Its advantage is the smallest reviewable change that can unblock Politics coverage, while its disadvantage is that it adds another activation surface that must be governed to avoid drift.

Option 3 is the right stage-one decision for this team and this risk profile. The immediate product bug is public Politics under-supply, and the immediate credibility bug is borrowed non-politics fallback content appearing in the Politics tab. A small manifest lets the public Home path include Reuters World while preserving backward compatibility for `getMvpDefaultPublicSources()` and direct callers of `generateDailyBriefing`. That matters because the audits identified several legitimate but disconnected runtime paths; collapsing them in one step would be larger than the current objective and harder to review safely.

The accepted consequence is that the manifest is now a real activation surface. Any future public source activation that changes what unauthenticated users see must be reviewed against this manifest, not only against the catalog, donor defaults, or MVP constants. The manifest should stay small, ordered, and covered by tests that fail when a declared source ID is missing from `demoSources`.

This decision should be revisited when the product adds a second public surface, when the team grows enough that source governance needs stronger ownership boundaries, or when the manifest grows beyond about 10 entries. At that point, the system should likely move toward the fuller runtime source-plan resolver described as Option 1 in the architecture audit.
