## Executive Summary

The public Home route calls `getDashboardPageState("/")`, then builds the homepage view model from the returned dashboard data; the public unauthenticated path inside `getDashboardData` calls `getPipelineBackedDashboardData`, which supplies `getMvpDefaultPublicSources()` into `generateDailyBriefing` (`src/app/page.tsx:15-24`, `src/lib/data.ts:254-280`, `src/lib/data.ts:568-580`). The public MVP default list is exactly `source-verge`, `source-ars`, `source-tldr-tech`, `source-techcrunch`, and `source-ft`, so the public Home/Politics category path does not include `horizon-reuters-world` or any other donor-default politics source (`src/lib/demo-data.ts:117-123`, `src/lib/demo-data.ts:125-136`). The no-argument pipeline path is a separate runtime path: `ingestRawItems()` without supplied sources resolves `DEFAULT_DONOR_FEED_IDS` plus `PROBATIONARY_RUNTIME_FEED_IDS`, including `horizon-reuters-world` and `mit-technology-review` (`src/lib/pipeline/ingestion/index.ts:67-90`, `src/adapters/donors/registry.ts:765-774`). Category routing is downstream content classification, not category-specific source selection: homepage sections are created from `HOMEPAGE_CATEGORY_CONFIG`, events are included when their `classification.primaryCategory` matches the section, and classification scores topic names, headlines, summaries, matched keywords, ranking signals, and source names (`src/lib/homepage-model.ts:175-220`, `src/lib/homepage-taxonomy.ts:28-123`, `src/lib/homepage-taxonomy.ts:147-183`). Recommendation: adopt a single runtime source-plan resolver that preserves the catalog as inventory, emits category-aware source subsets, and feeds both public and no-argument ingestion without rewriting the pipeline.

## Category Mapping Analysis

There is a centralized homepage category taxonomy for display categories: `HomepageCategoryKey` is exactly `"tech" | "finance" | "politics"`, and `HOMEPAGE_CATEGORY_CONFIG` defines labels, aliases, and keywords for all three (`src/lib/homepage-taxonomy.ts:3-11`, `src/lib/homepage-taxonomy.ts:28-123`). The code uses `finance` as the implementation key for the user-facing Economics/Finance category, and the finance aliases include `economics` and `macro` (`src/lib/homepage-taxonomy.ts:54-86`). The code does not declare a centralized source-to-category activation map for runtime public source selection; `countSourcesByHomepageCategory` classifies source `topicName` and source `name` using the same text classifier rather than looking up a source ID map (`src/lib/homepage-taxonomy.ts:185-204`).

Category assignment for public pipeline briefing items happens downstream from ingestion: `generateDailyBriefing` first runs the cluster-first pipeline with a supplied source list, then `resolvePipelineTopic` calls `classifyHomepageCategory` using cluster/intelligence text, keywords, ranking signals, and source names (`src/lib/data.ts:754-758`, `src/lib/data.ts:868-887`). The classifier chooses a primary category only when the highest category score is at least 4, and it can also report secondary categories when other scores are close enough (`src/lib/homepage-taxonomy.ts:147-183`). The homepage display then places events into category sections only when `event.classification.primaryCategory` equals the section key; an item with a secondary category is not independently inserted into that secondary section by the current section-selection code (`src/lib/homepage-model.ts:175-181`).

Signed-in personalized category matching also happens downstream from source selection: `syncTopicMatches` reads stored articles and matches each article to the user's topics by keyword text, and `getPrimaryMatches` later chooses the highest-scoring topic match for clustering and briefing assembly (`src/lib/data.ts:1011-1091`, `src/lib/topic-matching.ts:45-95`, `src/lib/data.ts:1929-1949`). Signed-in event clustering groups passed articles by primary matched topic ID before event clustering, so user topics can route articles into a topic even when the source itself was not selected by category (`src/lib/data.ts:1119-1153`, `src/lib/data.ts:1189-1200`). Default signed-in topics include `Tech`, `Finance`, `World`, and `Business`, with `World` carrying geopolitics/government keywords rather than a literal `Politics` topic name (`src/lib/default-topics.ts:3-67`).

The source catalog has per-source `topicLabel` metadata, but this metadata is not runtime activation routing in the traced public path: `recommendedSources` is exported as a catalog array, while the public runtime path uses `getMvpDefaultPublicSources()` and the no-argument runtime path uses donor registry defaults and probationary feeds (`src/lib/source-catalog.ts:1-19`, `src/lib/data.ts:568-580`, `src/lib/pipeline/ingestion/index.ts:67-90`). Tests explicitly guard that the catalog has no `active_default` entries and that non-default demo/catalog entries do not become public default ingestion (`src/lib/source-defaults.test.ts:46-55`). The catalog therefore operates as human-readable inventory today, while runtime activation is split across MVP public IDs, donor default IDs, probationary IDs, and user Supabase sources (`src/lib/demo-data.ts:117-123`, `src/adapters/donors/registry.ts:765-774`, `src/lib/data.ts:283-360`).

## Politics Page Diagnosis

There is no dedicated file route for `/politics`; the Politics surface is a Home/briefing category tab produced from `categorySections`, and `/dashboard` redirects to `/` (`src/components/home/CategoryTabStrip.tsx:65-82`, `src/app/dashboard/page.tsx:1-5`). When an unauthenticated production user loads the public Home surface, `getDashboardData` takes the public branch and `getPipelineBackedDashboardData` supplies `getMvpDefaultPublicSources()` into `generateDailyBriefing` (`src/lib/data.ts:254-280`, `src/lib/data.ts:568-580`). The only source runtime IDs that can directly feed that public Politics tab through the current public source path are `source-verge`, `source-ars`, `source-tldr-tech`, `source-techcrunch`, and `source-ft`, because those are the exact MVP defaults resolved by `getMvpDefaultPublicSources()` (`src/lib/demo-data.ts:117-136`).

`horizon-reuters-world` is technically wired in the donor no-argument runtime: it is in `DEFAULT_DONOR_FEED_IDS`, Horizon defines it as active/default with topic `World`, and no-argument ingestion includes default donor feeds (`src/adapters/donors/registry.ts:765-770`, `src/adapters/donors/horizon/index.ts:17-36`, `src/lib/pipeline/ingestion/index.ts:67-90`). That wiring is a broken link for the public Politics surface because the public Home path passes supplied MVP sources into the pipeline, and supplied-source ingestion bypasses default/probationary donor resolution by filtering and mapping only the supplied `Source[]` (`src/lib/data.ts:568-580`, `src/lib/data.ts:754-758`, `src/lib/pipeline/ingestion/index.ts:67-96`). The same broken-link diagnosis applies to `mit-technology-review` for public Technology coverage: it is active/probationary in no-argument ingestion but bypassed by the supplied MVP public source path unless added to a supplied source list (`src/adapters/donors/registry.ts:772-774`, `src/adapters/donors/openclaw/index.ts:57-76`, `src/lib/pipeline/ingestion/index.ts:67-96`).

The public Politics tab can still display an item if one of the five MVP sources produces an article or cluster whose content scores as politics; politics scoring uses topic name, headline, summary, why-it-matters text, source names, matched keywords, and ranking signals, not source ID membership (`src/lib/homepage-taxonomy.ts:206-256`, `src/lib/data.ts:868-887`, `src/lib/homepage-model.ts:303-322`). If no current event classifies as politics, the section can borrow fallback events from other categories, because `allocateFallbackEvents` explicitly selects events whose primary category is not the empty category (`src/lib/homepage-model.ts:222-235`, `src/lib/homepage-model.ts:588-630`). Therefore the public Politics tab today can show true politics only from MVP-source content that classifies as politics, or non-politics fallback cards when the politics section is empty; it cannot show Reuters World articles through the public source path unless the public path is changed or production runtime behavior differs from the static code trace (`src/lib/demo-data.ts:117-136`, `src/lib/data.ts:568-580`, `src/lib/homepage-model.ts:588-630`).

Unknown requiring preview/log verification: this static audit cannot prove which production users are authenticated, which user-specific Supabase sources exist, or whether live feeds are currently reachable; those runtime facts determine signed-in personalized results and feed-success outcomes (`src/lib/data.ts:283-360`, `src/lib/data.ts:1608-1839`, `src/lib/rss.ts:25-67`). Unknown requiring preview/log verification: if production has environment variables or database rows that differ from local static assumptions, signed-in users may see custom politics-capable feeds through the personalized Supabase path even though unauthenticated public users cannot receive `horizon-reuters-world` from the public MVP path (`src/lib/data.ts:283-360`, `src/app/actions.ts:347-425`).

## Architectural Options

### Option 1 — Recommendation Candidate: Runtime Source Plan Resolver

Recommendation: create one runtime source-plan resolver that returns named source subsets for surfaces such as `public.home`, `public.category.tech`, `public.category.finance`, `public.category.politics`, `no_argument.runtime`, and `personalized.fallback`, while keeping `recommendedSources` as the inventory layer. Recommendation: this option keeps the donor registry as executable feed metadata, keeps the catalog as human-readable inventory, merges MVP defaults/donor defaults/probationary defaults into a single selection policy, and removes direct public calls to `getMvpDefaultPublicSources()` as the source-of-truth path. Recommendation: the estimated scope is moderate and likely touches `src/lib/demo-data.ts`, `src/lib/data.ts`, `src/lib/pipeline/ingestion/index.ts`, `src/adapters/donors/registry.ts`, source-default tests, homepage-model/data fallback tests, and a new focused resolver test file. Recommendation: the gain is a single explainable activation layer that can return different category subsets without hardcoding one global list; the loss is that the simple five-source MVP constant stops being the obvious public default. Recommendation: migration risk is highest around fallback semantics and tests that assert exact IDs, because current tests deliberately protect the five-source MVP set and no-argument donor set as separate facts.

### Option 2 — Recommendation Candidate: Donor Registry As Runtime Truth, Catalog As Metadata Overlay

Recommendation: make the donor registry the only executable source resolver and add surface/category activation metadata to donor feed definitions, while the catalog remains a read-only human inventory and validation-status reference. Recommendation: this option keeps `DEFAULT_DONOR_FEED_IDS` and `PROBATIONARY_RUNTIME_FEED_IDS` concepts but merges public default selection into donor feed metadata such as `surfaces`, `categoryEligibility`, and `progressionStage`. Recommendation: the estimated scope is smaller than Option 1 in ingestion code but broader in donor metadata and tests, likely touching donor definitions, registry selectors, `src/lib/data.ts`, and source-default tests. Recommendation: the gain is minimal pipeline disruption because no-argument ingestion already resolves donor feeds; the loss is that demo-source IDs such as `source-verge` and donor IDs such as `openclaw-the-verge` must be normalized or aliased carefully. Recommendation: migration risk centers on source identity drift, because public surfaces currently emit `Source` IDs while donor ingestion emits `SourceDefinition.sourceId`.

### Option 3 — Recommendation Candidate: Category Source Manifests On Top Of Existing Lists

Recommendation: add a small manifest that maps categories and surfaces to existing runtime IDs while leaving MVP defaults, donor defaults, and probationary defaults mostly intact. Recommendation: this option keeps current lists, adds a category/surface layer above them, and uses the manifest to assemble supplied source sets for Home and category paths. Recommendation: the estimated scope is the smallest initial change, likely one new manifest module plus changes in `src/lib/data.ts` and tests. Recommendation: the gain is fast unblocking for Politics because the manifest can explicitly add `horizon-reuters-world` to the public politics-eligible subset; the loss is that it adds a fifth place where source activation can drift unless the manifest becomes the enforced source-of-truth. Recommendation: migration risk is low for initial behavior but high for long-term governance because it preserves the split instead of eliminating it.

## Recommendation

Recommendation: choose Option 1, the Runtime Source Plan Resolver, as the primary direction. Recommendation: the evidence favors Option 1 because the current public path and no-argument path are both legitimate but disconnected: public Home supplies MVP sources directly, while no-argument ingestion resolves donor defaults and probationary feeds (`src/lib/data.ts:568-580`, `src/lib/pipeline/ingestion/index.ts:67-90`). Recommendation: the resolver should produce source plans rather than one global default list because the product requirement needs category subsets, and the current display classifier already supports multi-category treatment after ingestion through primary and secondary category scores (`src/lib/homepage-taxonomy.ts:147-183`). Recommendation: preserve `recommendedSources` as the inventory layer because current tests and catalog metadata already treat it as non-default human-readable inventory rather than runtime activation (`src/lib/source-catalog.ts:1-19`, `src/lib/source-defaults.test.ts:46-55`). Recommendation: avoid rewriting the ingestion pipeline because the existing pipeline already accepts supplied `Source[]` and has a no-argument donor resolver; the architectural fix is to unify source planning before those entry points, not replace ingestion, clustering, or ranking (`src/lib/data.ts:754-758`, `src/lib/pipeline/ingestion/index.ts:67-96`, `src/lib/pipeline/index.ts:28-45`).

## Migration Sequencing

Recommendation: Phase 1 should add read-only tests and a resolver snapshot that documents current source plans without changing behavior; include plans for public Home, Tech, Finance/Economics, Politics, no-argument runtime, and personalized fallback. Recommendation: Phase 2 should route public Home through the resolver while preserving the current MVP five-source output as the initial public home plan, so the first migration proves there is no public regression. Recommendation: Phase 3 should add category-aware public source plans, starting with Politics wiring for `horizon-reuters-world`, while keeping article/category display downstream and preserving multi-surface eligibility. Recommendation: Phase 4 should normalize identity reporting so equivalent public/demo and donor IDs can be audited without hiding their origin. Recommendation: Phase 5 should retire or downgrade direct use of `MVP_DEFAULT_PUBLIC_SOURCE_IDS` to a compatibility alias once resolver-backed tests prove the new source plans cover public, donor, probationary, and personalized fallback paths.
